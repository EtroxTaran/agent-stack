# PR Open

Open a pull request whose body is immediately usable by human reviewers
and the AI-review pipeline. The pipeline parses the AC-Verification
section to build coverage reports, so the format is not optional.

## Preconditions (fail closed)

1. `git rev-parse --show-toplevel` resolves (inside a repo).
2. `git status --porcelain` empty. Dirty tree -> abort.
3. Current branch is not `main`, `master`, or `develop`.
4. At least one commit exists between current branch and origin/main:
   `git rev-list --count origin/main..HEAD` > 0.
5. Current branch is pushed to origin (or will be pushed in Step 2).
6. `gh auth status` OK.

Any failure -> stop with an explicit message. Do not silently `--force`,
do not amend, do not skip hooks.

## Workflow

### Step 1 - Derive linked issue

From the current branch name, extract the issue number:

```bash
branch=$(git rev-parse --abbrev-ref HEAD)
# expected: feat/<slug>-issue-<N>, fix/<slug>-issue-<N>, etc.
N=$(echo "$branch" | grep -oE 'issue-[0-9]+' | grep -oE '[0-9]+$' || true)
```

- If `N` empty, ask the user for the issue number. Do not guess.
- If multiple `issue-N` tokens present (multi-issue PR), ask the user
  which one is primary - the rest go as `Refs #X` lines.

### Step 2 - Push branch if needed

```bash
git push -u origin "$branch" 2>/dev/null || git push
```

Only a normal push. No force. No force-with-lease. If push fails due to
diverged history, stop and report.

### Step 3 - Fetch issue context

```bash
gh issue view "$N" --json title,body,labels,url > /tmp/issue-$N.json
```

Parse the Gherkin block from `.body` - same helper as `issue-pickup`
skill's `scripts/parse_gherkin.py`. For each AC, locate the test file(s)
that reference it:

```bash
# heuristic: scan test files for the AC slug or AC number
grep -rln "AC-1\|<ac-slug>" -- 'tests/' '*.test.ts' '*.spec.ts' '**/test_*.py'
```

If no test file hits for an AC, mark it `Verified by: MISSING`. That
makes the gap visible in the PR body - do not hide it.

### Step 4 - Build PR title

Use Conventional Commits, derived from branch prefix:

- `feat/...` -> `feat: <issue-title-lowercased>`
- `fix/...` -> `fix: <issue-title-lowercased>`
- `refactor/...` -> `refactor: <issue-title-lowercased>`
- `chore/...` -> `chore: <issue-title-lowercased>`

Keep under 70 chars. Trim the issue title if necessary.

### Step 5 - Render PR body

Template:

```markdown
## Linked Issue

Closes #<N>
<Refs #X lines for secondary issues, one per line>

## Acceptance Criteria Verification

- [ ] **AC-1**: <scenario title from Gherkin>
      Verified by: `tests/path/to/ac_1.test.ts`
- [ ] **AC-2**: <scenario title>
      Verified by: `tests/path/to/ac_2.test.ts`

_Check each box only after the corresponding test is green on this branch._

## Test Plan

- Unit: `pnpm test --run` (or language equivalent)
- Typecheck: `pnpm typecheck`
- Lint: `pnpm lint`
- E2E: `pnpm test:e2e` - routes mocked via `page.route()` where external
  deps are involved
- Manual smoke: <describe only if automation cannot cover>

## Screenshots

<Only for UI changes. Omit section entirely otherwise.>

## Deployment Notes

<Migration, feature flag, env-var changes. Omit if none.>
```

Rules:
- No placeholder text left (`<...>` segments must be filled or sections
  removed).
- No "Generated by" boilerplate in the body - the reviewer pipeline adds
  its own footer.
- No emoji in title or body unless the user explicitly asks.

### Step 6 - Create PR

```bash
gh pr create \
  --base main \
  --head "$branch" \
  --title "$TITLE" \
  --body-file /tmp/pr-body-$N.md
```

For multi-line bodies with shell specials, always use `--body-file`, not
`--body "..."`.

### Step 7 - Report

```
PR opened: <url>
Linked issue: #<N>
AC coverage seeded: <filled>/<total> (<missing> marked MISSING)

Next steps:
- AI-review pipeline will run automatically on pull_request event.
- Ping the Discord project channel once consensus score is posted.
- Fill any AC-Verification checkboxes as you verify locally.
- If MISSING rows exist, add tests for those ACs before requesting human review.
```

## Error handling

- `gh pr create` fails with "A pull request already exists" - report the
  existing PR URL and stop. Do not open a duplicate.
- Branch name does not encode an issue - ask user, do not guess.
- Gherkin missing in issue - open the PR anyway with a `## Linked Issue`
  section and a plain "_No ACs declared in issue; please add before
  review._" note under `## Acceptance Criteria Verification`.

## Integration points

- Consumes output of `issue-pickup` (branch name convention).
- Feeds `ac-validate` skill (parses the AC-Verification table).
- Feeds AI-review pipeline Stage 5 (Gherkin-coverage check).
